Lichess backend is built with a highly modular architecture consisting of **83+ independent modules**, each focused on a specific domain or functionality. This modular design enables:

- **Separation of concerns** - Each module handles a specific aspect of the platform
- **Independent development** - Teams can work on different modules simultaneously
- **Code reusability** - Modules can depend on and use other modules
- **Maintainability** - Clear boundaries make the codebase easier to understand and maintain

## Module Structure

Each module follows a consistent directory structure:

```
modules/<module-name>/
├── src/
│   ├── main/
│   │   ├── *.scala          # Main implementation files
│   │   ├── Env.scala         # Module environment/configuration
│   │   ├── package.scala     # Package-level definitions
│   │   └── ui/               # UI components (if applicable)
│   └── test/
│       └── *Test.scala       # Unit tests
└── build.sbt                 # Module dependencies
```

## Module Dependencies

Modules are organized in a dependency hierarchy:

```scala
// Example from modules/game/src/main/Env.scala
final class Env(
    appConfig: Configuration,
    db: lila.db.Db,
    cacheApi: lila.memo.CacheApi,
    userApi: lila.user.UserApi,
    mongoCache: lila.memo.MongoCache.Api
)(using
    Executor,
    Scheduler,
    akka.stream.Materializer
):
  // Module initialization
```

### Common Dependency Patterns

1. **Core modules** (db, common, memo) - Depended on by most other modules
2. **Domain modules** (game, user, tournament) - Provide core business logic
3. **UI modules** (web, ui) - Depend on many domain modules
4. **Integration modules** (api, oauth) - Connect external services

## Module Communication

Modules communicate through:

### 1. Direct API Calls
```scala
// From modules/tournament/src/main/Env.scala
class Env(
    userApi: lila.user.UserApi,
    gameApi: lila.game.GameApi
):
  def createPairing(player1: UserId, player2: UserId) =
    for
      users <- userApi.byIds(List(player1, player2))
      game <- gameApi.start(users)
    yield game
```

### 2. Event Bus
```scala
// Publish events
bus.publish(Event.GameStart(gameId), 'gameStart)

// Subscribe to events
bus.subscribeFun('gameStart):
  case Event.GameStart(gameId) => handleGameStart(gameId)
```

### 3. Akka Streams
```scala
// Stream processing across modules
Source(gameIds)
  .mapAsync(4)(gameRepo.game)
  .collect { case Some(g) => g }
  .via(analysisFlow)
  .runWith(Sink.ignore)
```

## Best Practices

When working with the module system:

1. **Keep modules focused** - Each module should have a single, clear responsibility
2. **Minimize dependencies** - Depend only on what you need
3. **Use core types** - Leverage shared types from the `core` module
4. **Follow naming conventions** - Use consistent file and class naming
5. **Document APIs** - Public APIs should be well-documented
6. **Test independently** - Each module should have its own test suite
