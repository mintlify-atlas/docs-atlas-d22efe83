## Message Structure

All WebSocket messages follow a consistent JSON structure:

```typescript
interface Message {
  t: string;      // Message type (event name)
  d?: any;        // Data payload (optional)
  v?: number;     // Version number (optional, for versioned sockets)
}
```

## Client → Server Events

### Connection and Heartbeat

#### Ping

Keeps connection alive and measures latency.

```json
// Authenticated ping with lag
{
  "t": "p",
  "l": 5  // Average lag in deciseconds
}

// Simple ping
null
```

**Frequency:** Every 2.5 seconds

**Response:** Pong message (`0` or `{"t":"n"}`)

### Game Events

#### Move

Submits a chess move in a live game.

```json
{
  "t": "move",
  "d": {
    "u": "e2e4",     // UCI move notation
    "a": 123,        // Ack ID (if ackable)
    "l": 42,         // Client lag (if withLag)
    "s": "1a"        // Milliseconds encoded (if provided)
  }
}
```

**Options:**
- `ackable: true` - Requires server acknowledgment
- `withLag: true` - Include client lag in payload
- `sign: <signature>` - Required signature for validation

#### More Time

Requests additional time (in games that support it).

```json
{
  "t": "moretime"
}
```

Throttled to once per 300ms.

#### Flag

Claims timeout victory when opponent's clock expires.

```json
{
  "t": "flag",
  "d": "white"  // Color that ran out of time
}
```

Uses exponential backoff (500ms * 1.1^n) to avoid spam.

#### Berserk

Activates berserk mode in arena tournaments.

```json
{
  "t": "berserk",
  "d": {
    "a": 124  // Ack ID
  }
}
```

Throttled to once per 200ms, requires acknowledgment.

## Server → Client Events

### Connection Events

#### Pong

Response to ping, confirms connection is alive.

```
0  // Simple pong (single character)
```

Or:

```json
{
  "t": "n"  // Named pong
}
```

#### Resync

Instructs client to reload the page due to synchronization issues.

```json
{
  "t": "resync"
}
```

Client reloads after 500ms delay.

Location: `/home/daytona/workspace/source/ui/lib/src/socket.ts:294`

#### Acknowledgment

Confirms receipt of an ackable message.

```json
{
  "t": "ack",
  "d": 123  // Ack ID being confirmed
}
```

Client removes message from resend queue upon receiving ack.

#### Batch

Multiple events sent together.

```json
{
  "t": "batch",
  "d": [
    { "t": "event1", "d": "data1" },
    { "t": "event2", "d": "data2" }
  ]
}
```

Each sub-event processed individually.

## Message Helpers

### Creating Messages

Scala helper for creating JSON messages:

```scala
def makeMessage[A](t: String, data: A)(using writes: Writes[A]): JsObject =
  JsObject(Map("t" -> JsString(t), "d" -> writes.writes(data)))
```

Location: `/home/daytona/workspace/source/modules/core/src/main/socket.scala:76`

TypeScript equivalent:

```typescript
const msg = { t: "move", d: moveData };
```

### Ackable Messages

Messages requiring acknowledgment include an ack ID:

```typescript
// Client adds 'a' field
{
  "t": "move",
  "d": {
    "u": "e2e4",
    "a": 123  // Ack ID
  }
}

// Server responds
{
  "t": "ack",
  "d": 123
}
```

Location: `/home/daytona/workspace/source/ui/lib/src/socket.ts:386`

**Resend behavior:**
- Client resends every 1.2 seconds if not acknowledged
- Messages older than 2.5 seconds are resent
- Removed from queue upon receiving ack

## Error Handling

### Connection Errors

On WebSocket error or close:
1. Client marks as offline
2. Schedules reconnection (3.5s default)
3. Rotates to next socket domain if needed
4. Resends pending ackable messages after reconnection

Location: `/home/daytona/workspace/source/ui/lib/src/socket.ts:336`

### Version Gaps

If client detects missed events (version gap):
1. Retry handling message up to 10 times with 200ms delay
2. If still gapped after retries, reload page to resync
