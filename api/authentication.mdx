## Overview

Lichess uses OAuth2 for API authentication. OAuth allows your application to access Lichess on behalf of a user without handling their password.

## OAuth Flow Types

Lichess supports two OAuth2 flows:

1. **Authorization Code Flow** - For web applications
2. **PKCE Flow** - For mobile and native applications (recommended)

## OAuth Scopes

Scopes define what your application can access. Available scopes include:

| Scope | Description |
|-------|-------------|
| `preference:read` | Read user preferences |
| `preference:write` | Modify user preferences |
| `email:read` | Read user email address |
| `challenge:read` | Read incoming challenges |
| `challenge:write` | Create and manage challenges |
| `challenge:bulk` | Create bulk pairings |
| `study:read` | Read private studies |
| `study:write` | Create and modify studies |
| `tournament:write` | Create and manage tournaments |
| `racer:write` | Create puzzle races |
| `puzzle:read` | Read puzzle activity |
| `puzzle:write` | Submit puzzle attempts |
| `team:read` | Read team information |
| `team:write` | Join and manage teams |
| `team:lead` | Manage team as leader |
| `follow:read` | Read followed users |
| `follow:write` | Follow and unfollow users |
| `msg:write` | Send private messages |
| `board:play` | Play games via Board API |
| `bot:play` | Play games as a bot |
| `engine:read` | Read external engine configs |
| `engine:write` | Create external engines |
| `web:mod` | Moderator actions |
| `web:mobile` | Mobile app features |

## Authorization Code Flow

### Step 1: Direct User to Authorization URL

Redirect the user to:

```
https://lichess.org/oauth/authorize?
  client_id={client_id}&
  redirect_uri={redirect_uri}&
  response_type=code&
  code_challenge_method=S256&
  code_challenge={code_challenge}&
  scope={scopes}&
  state={state}
```

<ParamField path="client_id" type="string" required>
  Your application's client ID from OAuth app registration
</ParamField>

<ParamField path="redirect_uri" type="string" required>
  Where to redirect after authorization (must match registered URI)
</ParamField>

<ParamField path="response_type" type="string" required>
  Must be `code`
</ParamField>

<ParamField path="code_challenge_method" type="string">
  PKCE challenge method, use `S256` (SHA-256)
</ParamField>

<ParamField path="code_challenge" type="string">
  PKCE code challenge derived from code verifier
</ParamField>

<ParamField path="scope" type="string" required>
  Space-separated list of requested scopes
</ParamField>

<ParamField path="state" type="string">
  Random string to prevent CSRF attacks
</ParamField>

### Step 2: User Authorizes Application

The user will see an authorization screen showing:
- Your application name
- Requested permissions (scopes)
- Option to authorize or deny

### Step 3: Receive Authorization Code

After authorization, Lichess redirects to your `redirect_uri` with:

```
https://your-app.com/callback?
  code={authorization_code}&
  state={state}
```

### Step 4: Exchange Code for Access Token

<CodeGroup>
```bash cURL
curl -X POST https://lichess.org/api/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "code={authorization_code}" \
  -d "code_verifier={code_verifier}" \
  -d "redirect_uri={redirect_uri}" \
  -d "client_id={client_id}"
```

```javascript JavaScript
const response = await fetch('https://lichess.org/api/token', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/x-www-form-urlencoded',
  },
  body: new URLSearchParams({
    grant_type: 'authorization_code',
    code: authorizationCode,
    code_verifier: codeVerifier,
    redirect_uri: redirectUri,
    client_id: clientId,
  }),
});

const tokens = await response.json();
```

```python Python
import requests

response = requests.post('https://lichess.org/api/token', data={
    'grant_type': 'authorization_code',
    'code': authorization_code,
    'code_verifier': code_verifier,
    'redirect_uri': redirect_uri,
    'client_id': client_id,
})

tokens = response.json()
```
</CodeGroup>

## Using Access Tokens

Include the access token in the `Authorization` header:

```bash
curl https://lichess.org/api/account \
  -H "Authorization: Bearer {access_token}"
```

## Personal Access Tokens

For personal scripts or testing, you can create a Personal Access Token:

1. Go to https://lichess.org/account/oauth/token
2. Click "Create a new personal access token"
3. Select required scopes
4. Copy the generated token

<Warning>
  Personal access tokens have full access to your account. Never share them or commit them to version control.
</Warning>

## Token Management

### Revoke Token

<CodeGroup>
```bash cURL
curl -X DELETE https://lichess.org/api/token \
  -H "Authorization: Bearer {access_token}"
```
</CodeGroup>

Returns `204 No Content` on success.

### List Active Tokens

View all active OAuth tokens at: https://lichess.org/account/oauth/token

## Legacy OAuth (Deprecated)

The older OAuth flow using `POST /oauth` is deprecated. Use the standard OAuth2 endpoints instead:

- Authorization: `GET /oauth/authorize`
- Token exchange: `POST /api/token`

## Testing Tokens

### Test Token Validity

```bash
curl https://lichess.org/api/account \
  -H "Authorization: Bearer {token}"
```

If the token is valid, you'll receive account information. Invalid tokens return `401 Unauthorized`.

### Special Test Endpoints

<CodeGroup>
```bash Admin Challenge Tokens
curl -X POST https://lichess.org/api/token/admin-challenge \
  -H "Authorization: Bearer {admin_token}"
```

```bash Test Tokens
curl -X POST https://lichess.org/api/token/test \
  -H "Authorization: Bearer {token}"
```
</CodeGroup>

## Bot Accounts

For bot accounts, special requirements apply:

1. Account must be upgraded to bot account
2. Use `bot:play` scope for game play
3. Access bot-specific endpoints under `/api/bot/`

See the [Bot API documentation](https://lichess.org/api#tag/Bot) for details.

## Error Responses

### Authorization Errors

<ResponseExample>
```json Invalid Client

  "error_description": "Invalid client credentials"
}
```
</ResponseExample>

<ResponseExample>
```json Invalid Grant
{
  "error": "invalid_grant",
  "error_description": "Authorization code expired or invalid"
}
```
</ResponseExample>

### Common Error Codes

- `invalid_request` - Malformed request
- `invalid_client` - Invalid client ID
- `invalid_grant` - Invalid or expired authorization code
- `unauthorized_client` - Client not authorized for this grant type
- `unsupported_grant_type` - Grant type not supported
- `invalid_scope` - Invalid or excessive scope requested

## Security Best Practices

<AccordionGroup>
  <Accordion title="Always Use PKCE">
    The PKCE extension prevents authorization code interception attacks. Always use `code_challenge` and `code_verifier`.
  </Accordion>
  
  <Accordion title="Validate State Parameter">
    Always verify the `state` parameter matches what you sent to prevent CSRF attacks.
  </Accordion>
  
  <Accordion title="Store Tokens Securely">
    Never expose access tokens in client-side code or URLs. Store them securely on your backend.
  </Accordion>
  
  <Accordion title="Request Minimal Scopes">
    Only request the scopes your application actually needs. Users are more likely to authorize limited access.
  </Accordion>
</AccordionGroup>
