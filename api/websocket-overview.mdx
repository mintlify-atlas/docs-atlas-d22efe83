## Architecture

Lichess uses a separate WebSocket server called **lila-ws** to handle real-time communication. This architecture separates WebSocket connections from the main Lila application server, allowing for better scalability and performance.

### Key Components

- **lila-ws** - Dedicated WebSocket server (separate repository: [lichess-org/lila-ws](https://github.com/lichess-org/lila-ws))
- **Redis** - Message broker for communication between Lila and lila-ws
- **Lila** - Main application server that publishes messages via Redis
- **Client** - Browser-based TypeScript client that connects via WebSocket

### Communication Flow

```
Client (Browser) <--WebSocket--> lila-ws <--Redis--> Lila (Main Server)
```

## Connection Setup

### Client-Side Connection

The client establishes a WebSocket connection to the lila-ws server. Connection URLs follow this pattern:

```
ws[s]://<socket-domain>/<endpoint>?sri=<sri>&v=<version>&from=website
```

#### Connection Parameters

- **sri** (Socket Random ID) - Unique identifier for the socket connection, generated client-side
- **v** (version) - Socket version number for event ordering and synchronization
- **from** - Connection source identifier (typically "website")

### Socket Random ID (SRI)

The SRI is a unique identifier generated by each client when initiating a connection:

```typescript
// Generated using crypto.getRandomValues() or Math.random() fallback
const sri = randomToken(); // Example: "abc123def456"
```

**Purpose:**
- Uniquely identifies each socket connection
- Allows multiple connections from the same user
- Used for routing messages to specific connections
- Prevents processing duplicate events from the same client

## Connection Management

### Auto-Reconnection

The client automatically handles reconnection with exponential backoff:

```typescript
{
  pongTimeout: 9000,        // Wait 9s for pong before reconnecting
  pingDelay: 2500,          // Send ping every 2.5s
  autoReconnectDelay: 3500  // Wait 3.5s before reconnecting
}
```

Location in source: `/home/daytona/workspace/source/ui/lib/src/socket.ts:117`

### Ping/Pong Mechanism

The client maintains connection health through a ping/pong system:

1. Client sends ping every 2.5 seconds
2. Server responds with pong (message type `0` or `{"t":"n"}`)
3. If no pong received within 9 seconds, reconnect
4. Tracks average lag based on round-trip time

```typescript
// Ping message
{ "t": "p", "l": 5 }  // l = average lag in deciseconds

// Simple ping
null
```

### Version Synchronization

Many sockets use version numbers to ensure event ordering:

- Each event has a version number `v`
- Client tracks the current version
- If client receives `v > current + 1`, indicates missed events
- Client may retry or reload to sync state

### Protocol Layer

Messages between Lila and lila-ws use a text-based protocol:

```scala
// Incoming format: "<command> <args>"
"connect/user bob"
"disconnect/users alice,charlie"
"tell/user alice {\"t\":\"message\",\"d\":\"data\"}"
```

## Message Format

All WebSocket messages use JSON with a standardized structure:

```typescript
{
  "t": string,      // Message type
  "d": any,         // Optional data payload
  "v": number       // Optional version number
}
```

See [WebSocket Events](/api/websocket-events) for detailed message types.

## Connection Security

### Authentication

Authenticated connections include:
- User session validation on connection
- Signed messages for critical actions (moves, etc.)
- CSRF protection for state-changing operations

### Message Signing

Critical messages (game moves, racer scores) require signing:

```typescript
socket.send('move', moveData, { 
  sign: currentSign,  // Must match server-expected signature
  ackable: true       // Requires acknowledgment
});
```

## Idle and Disconnection Handling

### Idle Detection

After 10 minutes of user inactivity:
- Socket marked as idle
- Reconnection delays increased
- After 2 hours idle, connection destroyed

Location: `/home/daytona/workspace/source/ui/lib/src/socket.ts:354`

### Graceful Shutdown

During server maintenance:
1. Lila sends stop request to lila-ws
2. Active connections gracefully close
3. Clients automatically reconnect to healthy servers

## Multiple Socket Domains

Lichess supports multiple WebSocket domains for redundancy:

```typescript
const baseUrls = document.body.dataset.socketDomains.split(',');
// Example: ["socket1.lichess.org", "socket2.lichess.org"]
```

On connection failure, client rotates through available domains.

## Debugging

Enable debug logging:

```javascript
// In browser console
lichess.socket.options.debug = true;
```

This logs all sent/received messages for troubleshooting.
